# お題生成スキル

タイピングゲーム用のお題（`Sentence[]`）を複数のサブエージェントで並列生成し、検証して `src/lib/sentences.toml` に追記する。

## 実行環境の前提

必要なツール: `Read` / `Write` / `Bash` / `Agent`。**Agent ツールが使えない場合は処理を中断し、ユーザーに「Agent ツールが必要です」と報告して終了する。**

cwd は **リポジトリルート**（`package.json` がある階層）で実行する。`scripts/validate-sentences.ts` は相対パスで参照される。

## ステップ 1: ユーザーへのヒアリング

以下を確認する（まとめて一度に聞く）：

1. **生成数**: 合計何文生成するか（例: 10、20、50）
2. **N**: 1サブエージェントあたりの最大生成数（例: 10）。サブエージェント数は `ceil(生成数 / N)` になる
3. **長さ**: 短め（kana 15文字以内）/ 普通（16〜22文字）/ 長め（23〜29文字）
4. **テーマ**: 任意（例: 日常生活、自然、食べ物、スポーツ）。空欄でもよい
5. **出力先**: TOMLファイルパス（デフォルト: `src/lib/sentences.toml`）

## ステップ 2: お題生成（claude-haiku サブエージェントを並列起動）

`ceil(生成数 / N)` 個のサブエージェントを **1つのメッセージで同時起動**する（並列実行）。

- エージェント i（1始まり）が担当する件数: `min(N, 残り件数)`
- 各エージェントの一時ファイルパス: `/tmp/sentences_draft_<i>.toml`

```
Agent(
  description: "Generate typing sentences (batch <i>/<total>)",
  subagent_type: "general-purpose",
  model: "haiku",
  prompt: "<下記の生成プロンプト本文（プレースホルダー置換済み）>"
)
```

サブエージェントには **「TOML 形式のみを返し、説明文・コードフェンスは付けないこと」** を生成プロンプト末尾に必ず含めて指示する。出力にコードフェンスや説明文が混入したら、**最初の `[[sentences]]` から末尾までを切り出す**（それ以外は捨てる）。

### 生成プロンプト本文

```
あなたはタイピングゲーム用のお題を生成するアシスタントです。
以下のルールに厳密に従って、<COUNT>文のお題を TOML 形式で出力してください。
**返答は TOML のみ**。説明文・コードフェンス（` ``` `）は付けないでください。

## 出力形式

[[sentences]]
jp = "表示用テキスト"
kana = "打鍵列"
romaji = "ローマ字表示"

[[sentences]]
jp = "..."
kana = "..."
romaji = "..."

`romaji` は `kana` をローマ字読みしたもの（ヒント表示用）。
各エントリの間には空行を1行入れること。

## kana フィールドの絶対ルール

1. **漢字・カタカナは禁止**。必ずひらがなに変換する
   - NG: `kana = "東京"` / OK: `kana = "とうきょう"`
   - NG: `kana = "コーヒー"` / OK: `kana = "こーひー"`
2. **英単語は小文字のみ**。大文字禁止
   - NG: `kana = "USB接続"` / OK: `kana = "usbせつぞく"`
   - NG: `kana = "Wi-Fi"` / OK: `kana = "wi-fiがつながらない"`
3. **長音符 `ー` はひらがなと共存 OK**
   - `こーひー`、`すぽーつ` のように使う
4. **完全英語文は禁止**。必ず日本語を含む
5. **鍵カッコ「」『』は禁止**
6. **kana の長さは最大 29 文字**（JavaScript の `s.length`（UTF-16 code unit）で数える。`ー` `・` 半角英数も 1 文字）
7. **スペース禁止**
8. **全角英数字禁止**（半角のみ）

## 数字・英字混在の例

| jp | kana |
|---|---|
| USB接続する | usbせつぞくする |
| 3時間勉強した | 3じかんべんきょうした |
| Wi-Fiが繋がらない | wi-fiがつながらない |

## ん のルール

- `ん` の次が母音・y・n・語末の場合 → `nn` で打つ必要があるが、kana としては `ほんをよむ` と書けばエンジンが自動判定する

## テーマ

## チェックリスト（出力前に各文を確認）

- [ ] kana に漢字が含まれていないか

**プレースホルダーの置き換え:**
- `<COUNT>` → そのサブエージェントが担当する件数（`min(N, 残り件数)`）
- `<LENGTH_INSTRUCTION>` → 以下のいずれか（範囲はすべて両端を含む / `s.length` で計測）:
  - 短め: `kana の長さ s.length は 1〜15 にしてください（両端を含む）`
  - 普通: `kana の長さ s.length は 16〜22 にしてください（両端を含む）`
  - 長め: `kana の長さ s.length は 23〜29 にしてください（両端を含む）`
- `<THEME_INSTRUCTION>` → ユーザーが指定したテーマ（未指定なら `日常的な日本語の文にしてください`）

全サブエージェントの出力（TOML文字列）を受け取ったら、ステップ 3 へ。

## ステップ 3: 一時ファイル保存 → TOML→JSON変換 → 個別バリデーション

サブエージェント i の結果について以下を順に実行する（i=1, 2, … を順番に処理する）。

### 3-a. TOML を一時ファイルに保存

サブエージェント i の出力 TOML を `/tmp/sentences_draft_<i>.toml` に書き出す。

**フォールバック**: `/tmp` への書き込みが `permission denied` 等で弾かれた場合のみ、出力先と同じディレクトリに `.sentences_draft_<i>.toml` を置く。事前に環境を判定せず、まず `/tmp` を試して失敗してから切り替える。

### 3-b. TOML → JSON 変換

```bash
node --input-type=module -e "
import { readFileSync, writeFileSync } from 'fs';
import { parse } from 'smol-toml';
const { sentences } = parse(readFileSync('/tmp/sentences_draft_<i>.toml', 'utf-8'));
writeFileSync('/tmp/sentences_draft_<i>.json', JSON.stringify(sentences));
"
```

（フォールバックパスを使った場合はパスを適宜読み替える）

### 3-c. 検証スクリプトを実行

```bash
node --experimental-strip-types scripts/validate-sentences.ts /tmp/sentences_draft_<i>.json
```

Bash ツールは exit code ≠ 0 のときエラーとして表示される（`Error: ... (exit code N)` の形）。エラー表示が無ければ `0`、有る場合は表示された数値を見る。`echo $?` 等の後続コマンドは不要。

スクリプトの終了コードを確認する：
- `0` → 全文有効
- `1` (type-A エラーのみ) → type-A エラー文を除外して有効文のみ残す
- `2` (type-B エラーのみ) → type-B エラーについて GitHub Issue を登録（ステップ 5）し、有効文は残す
- `3` (両方) → type-A 除外 + type-B Issue 登録の両方を行い、有効文は残す

各バッチの有効文リストを累積する。全バッチの処理が終わったらステップ 4 へ。

## ステップ 4: 有効な文を sentences.toml に追記し、JSON を再生成する

全バッチの有効文を結合して、ユーザーが指定した出力先（TOML）に**追記**する。

手順：
1. 出力先ファイルが**すでに存在する場合**は、`[[sentences]]` の出現回数を数えて既存件数とする
2. 全バッチの有効文を以下の TOML 形式に変換する：
   ```toml

   [[sentences]]
   jp = "<jp>"
   kana = "<kana>"
   romaji = "<romaji>"
   ```
   （各エントリの前に空行を1行入れる）
3. ファイルの末尾に追記する（既存内容は変更しない）
4. 追記後、`npm run gen` を実行して `src/lib/generated/sentences.json` を再生成する

ファイルに書き出し後、結果をユーザーに報告する：
- サブエージェント数 / 合計生成数 / 合計有効数 / type-A 除外数 / type-B Issue 登録数
- 追記前の既存件数 / 追記後の合計件数
- 出力先パス（TOML）および再生成された JSON パス

## ステップ 5: type-B エラーの GitHub Issue 登録

type-B エラー（エンジン側のバグ）が存在する場合、各エラーについて GitHub Issue を登録する。

**リポジトリ: `ahy4/typing`**

Issue の内容：
- **タイトル**: `[typing-engine] parseKana が処理できない kana パターン: "<kana文字列>"`
- **本文**:
  ```
  ## 概要
  お題生成スキルの検証で、有効な kana 文字列を parseKana() が正しく処理できないケースを検出しました。

  ## 再現情報
  - jp: <japanese文字列>
  - kana: <kana文字列>
  - エラー詳細: <スクリプトが出力したエラー理由>

  ## 期待される動作
  上記 kana を parseKana() に渡すと、全セグメントに有効な romaji オプションが返るべきです。

  ## 発生環境
  scripts/validate-sentences.ts (type-B error)
  ```
