# お題生成スキル

タイピングゲーム用のお題（`Sentence[]`）を並列生成・多段検証し、`src/lib/sentences.toml` に追記する。

## 実行環境の前提

必要なツール: `Read` / `Write` / `Bash` / `Agent`。**Agent ツールが使えない場合は処理を中断し、ユーザーに「Agent ツールが必要です」と報告して終了する。**

cwd は **リポジトリルート**（`package.json` がある階層）で実行する。`scripts/validate-sentences.ts` は相対パスで参照される。

## ステップ 1: ユーザーへのヒアリング

以下を確認する（まとめて一度に聞く）：

1. **生成数**: 合計何文生成するか（例: 10、20、50）
2. **N**: 1サブエージェントあたりの最大生成数（例: 10）。サブエージェント数は `K = ceil(生成数 / N)` になる
3. **JP の長さ**: 短め（jp 10文字以内）/ 普通（jp 11〜18文字）/ 長め（jp 19〜25文字）
   - `jp.length`（UTF-16 code unit）で計測。kana の長さは聞かない
4. **テーマ**: 任意（例: 日常生活、自然、食べ物、スポーツ）。空欄でもよい
5. **出力先**: TOMLファイルパス（デフォルト: `src/lib/sentences.toml`）

## ステップ 2: お題生成（claude-haiku サブエージェントを並列起動）

`K = ceil(生成数 / N)` 個のサブエージェントを **1つのメッセージで同時起動**する（並列実行）。

- エージェント i（1始まり）が担当する件数: `min(N, 残り件数)`

```
Agent(
  description: "Generate typing sentences (batch <i>/<K>)",
  subagent_type: "general-purpose",
  model: "haiku",
  prompt: "<下記の生成プロンプト本文（プレースホルダー置換済み）>"
)
```

出力にコードフェンスや説明文が混入したら、**最初の `[[sentences]]` から末尾までを切り出す**（それ以外は捨てる）。

### 生成プロンプト本文

```
あなたはタイピングゲーム用のお題を生成するアシスタントです。
以下のルールに厳密に従って、<COUNT>文のお題を TOML 形式で出力してください。
**返答は TOML のみ**。説明文・コードフェンス（` ``` `）は付けないでください。

## 出力形式

[[sentences]]
jp = "表示用テキスト（漢字混じり）"
kana = "打鍵列（ひらがな・英数字）"

[[sentences]]
jp = "..."
kana = "..."

各エントリの間には空行を1行入れること。

## jp と kana の対応関係（最重要）

`jp` は表示テキスト（漢字混じり）、`kana` は jp を**全文発音した**打鍵列です。

**kana は jp の完全な読みでなければならない。絶対に途中で切らない。**

jp のすべての文字に対応する読みが kana に含まれていること:
- 漢字 → 正しいひらがな読み
- ひらがな → そのまま
- カタカナ → ひらがなに変換（例: コーヒー → こーひー）
- 英字 → 小文字のまま（例: USB → usb）
- 数字 → そのまま（例: 3 → 3）

jp に漢字があれば kana は必ず jp より文字数が多くなる。
kana の文字数が jp の文字数以下の場合は kana が途中で切れている証拠。

### 対応例

| jp | kana |
|---|---|
| 今日はいい天気だ | きょうはいいてんきだ |
| コーヒーを飲む | こーひーをのむ |
| USB接続する | usbせつぞくする |
| 3時間勉強した | 3じかんべんきょうした |

### NG の例（禁止）

```
NG: jp を書いてから kana を短く切る
  jp = "毎朝のコーヒーは欠かせない習慣です"
  kana = "まいあさのこーひーはかかせない"  ← 「しゅうかんです」がない！

OK: kana が jp の完全な読み
  jp = "毎朝のコーヒーは欠かせない習慣です"
  kana = "まいあさのこーひーはかかせないしゅうかんです"
```

## jp フィールドのルール

1. **長さ**: <JP_LENGTH_INSTRUCTION>
2. **自然な日本語文**であること（意味が通る文）
3. **鍵カッコ「」『』は禁止**

## kana フィールドの絶対ルール

1. **漢字・カタカナは禁止**。必ずひらがなに変換する
   - NG: `kana = "東京"` / OK: `kana = "とうきょう"`
   - NG: `kana = "コーヒー"` / OK: `kana = "こーひー"`
2. **英単語は小文字のみ**。大文字禁止
   - NG: `kana = "USB接続"` / OK: `kana = "usbせつぞく"`
3. **長音符 `ー` はひらがなと共存 OK**（`こーひー`、`すぽーつ`）
4. **完全英語文は禁止**。必ず日本語を含む
5. **鍵カッコ「」『』は禁止**
6. **スペース禁止**
7. **全角英数字禁止**（半角のみ）

## ん のルール

`ん` の次が母音・y・n・語末の場合 → kana としては `ほんをよむ` のように書けばエンジンが自動判定する

## テーマ

## ステップ 3: JP-kana 対応検証（各バッチに 2 つの検証サブエージェントを並列起動）

**目的**: 生成モデルが jp の途中で kana を切っていないか、誤読していないかを LLM で意味的に確認する。

各バッチ i について 2 つの検証サブエージェントを起動する。K バッチあれば **2K 個のサブエージェントを 1 つのメッセージで同時起動**する。

```
Agent(
  description: "Validate jp-kana (batch <i>, validator <1 or 2>)",
  subagent_type: "general-purpose",
  model: "haiku",
  prompt: "<下記の検証プロンプト本文（プレースホルダー置換済み）>"
)
```

### 検証プロンプト本文

```
あなたは日本語タイピングゲーム用のお題データを検証するアシスタントです。
以下の文リストについて、各文の `kana` が `jp` の**正確かつ完全な**読みになっているかを判定してください。

## 判定基準

各文について次をすべて確認する：

1. **完全性**: kana は jp の最初から最後まで全文字を読んでいるか（途中で切れていないか）
2. **漢字の読み**: jp の漢字が正しいひらがな読みで kana に含まれているか
3. **ひらがなの一致**: jp のひらがなはそのまま kana に含まれているか
4. **英数字の一致**: jp の英字（大文字も）が小文字で kana に含まれているか、数字がそのまま含まれているか
5. **カタカナの変換**: jp のカタカナがひらがなに変換されて kana に含まれているか（ー は ー のまま OK）

## NG の例

| jp | kana | 理由 |
|---|---|---|
| 毎朝のコーヒーは欠かせない習慣です | まいあさのこーひーはかかせない | 「習慣です」→「しゅうかんです」が欠落 |
| USB接続がうまくいかない | usbせつぞくがうまく | 「いかない」が欠落 |
| 桜が満開になる季節は美しい | さくらがまんかいになるきせつは | 「は美しい」→「はうつくしい」が欠落 |

## 出力形式

**JSON 配列のみを返してください。説明文・コードフェンスは付けないでください。**

[
  {"index": 0, "valid": true},
  {"index": 1, "valid": false, "reason": "「習慣です」に対応する「しゅうかんです」が kana に含まれていない"},
  ...
]

## 対象データ

<SENTENCES_JSON>
```

**プレースホルダーの置き換え:**
- `<SENTENCES_JSON>` → バッチ i の文を JSON 配列で渡す:
  ```json
  [{"index": 0, "jp": "今日はいい天気だ", "kana": "きょうはいいてんきだ"}, ...]
  ```

### 結果の統合ルール

バッチ i の文 j について:
- **両方の検証サブエージェントが `valid: true`** → ステップ 4 へ進む
- **どちらか一方でも `valid: false`** → 除外（LLM-reject として記録）

全バッチの有効文リストを累積したらステップ 4 へ。

## ステップ 4: フォーマット・エンジン検証（validate-sentences.ts）

ステップ 3 を通過した文をまとめて一時ファイルに保存し、エンジン検証を実行する。

### 4-a. JSON ファイルを保存

```bash
node --input-type=module -e "
import { writeFileSync } from 'fs';
const sentences = <有効文リストの JSON 文字列>;
writeFileSync('/tmp/sentences_to_validate.json', JSON.stringify(sentences));
"
```

### 4-b. 検証スクリプトを実行

```bash
node --experimental-strip-types scripts/validate-sentences.ts /tmp/sentences_to_validate.json
```

Bash ツールは exit code ≠ 0 のときエラーとして表示される（`Error: ... (exit code N)` の形）。エラー表示が無ければ `0`、有る場合は表示された数値を見る。

終了コードに応じた処理：
- `0` → 全文有効
- `1` (type-A エラーのみ) → type-A エラー文を除外して有効文のみ残す
- `2` (type-B エラーのみ) → type-B エラーについて GitHub Issue を登録（ステップ 6）し、有効文は残す
- `3` (両方) → type-A 除外 + type-B Issue 登録の両方を行い、有効文は残す

## ステップ 5: 有効な文を sentences.toml に追記し、JSON を再生成する

手順：
1. 出力先ファイルが**すでに存在する場合**は、`[[sentences]]` の出現回数を数えて既存件数とする
2. 全バッチの有効文を以下の TOML 形式に変換する：
   ```toml

   [[sentences]]
   jp = "<jp>"
   kana = "<kana>"
   ```
   （各エントリの前に空行を1行入れる）
3. ファイルの末尾に追記する（既存内容は変更しない）
4. 追記後、`npm run gen` を実行して `src/lib/generated/sentences.json` を再生成する

ファイルに書き出し後、結果をユーザーに報告する：
- 生成バッチ数 / 合計生成数 / LLM検証除外数 / type-A除外数 / type-B Issue登録数 / 合計有効数
- 追記前の既存件数 / 追記後の合計件数
- 出力先パス（TOML）および再生成された JSON パス

## ステップ 6: type-B エラーの GitHub Issue 登録

type-B エラー（エンジン側のバグ）が存在する場合、各エラーについて GitHub Issue を登録する。

**リポジトリ: `ahy4/typing`**

Issue の内容：
- **タイトル**: `[typing-engine] parseKana が処理できない kana パターン: "<kana文字列>"`
- **本文**:
  ```

  お題生成スキルの検証で、有効な kana 文字列を parseKana() が正しく処理できないケースを検出しました。

  ## 再現情報
  - jp: <japanese文字列>
  - kana: <kana文字列>
  - エラー詳細: <スクリプトが出力したエラー理由>

  ## 期待される動作
  上記 kana を parseKana() に渡すと、全セグメントに有効な romaji オプションが返るべきです。

  ## 発生環境
  scripts/validate-sentences.ts (type-B error)
  ```
