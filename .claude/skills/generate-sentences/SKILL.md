# お題生成スキル

タイピングゲーム用のお題（`Sentence[]`）を生成し、検証してJSONに書き出す。

## 実行環境の前提

必要なツール: `Read` / `Write` / `Bash` / `Agent`。**Agent ツールが使えない場合は処理を中断し、ユーザーに「Agent ツールが必要です」と報告して終了する。**

cwd は **リポジトリルート**（`package.json` がある階層）で実行する。`scripts/validate-sentences.ts` は相対パスで参照される。

## ステップ 1: ユーザーへのヒアリング

以下を確認する（まとめて一度に聞く）：

1. **生成数**: 何文生成するか（例: 10、20、50）
2. **長さ**: 短め（kana 15文字以内）/ 普通（16〜22文字）/ 長め（23〜29文字）
3. **テーマ**: 任意（例: 日常生活、自然、食べ物、スポーツ）。空欄でもよい
4. **出力先**: JSONファイルパス（デフォルト: `src/lib/generated/sentences.json`）

## ステップ 2: お題生成（claude-haiku サブエージェント）

Agent ツールで claude-haiku サブエージェントを起動する。

```
Agent(
  description: "Generate typing sentences",
  subagent_type: "general-purpose",
  model: "haiku",
  prompt: "<下記の生成プロンプト本文>"
)
```

サブエージェントには **「JSON 配列のみを返し、説明文・コードフェンスは付けないこと」** を生成プロンプト末尾に必ず含めて指示する。出力に説明文や ```json ... ``` フェンスが混入したら、**最初の `[` から対応する最後の `]` までを切り出して** JSON.parse にかける（行頭の説明・末尾の説明は捨てる）。

### 生成プロンプト本文

```
あなたはタイピングゲーム用のお題を生成するアシスタントです。
以下のルールに厳密に従って、<COUNT>文のお題を JSON 配列で出力してください。
**返答は JSON 配列のみ**。説明文・コードフェンスは付けないでください。

## 出力形式

```json
[
  { "jp": "表示用テキスト", "kana": "打鍵列", "romaji": "ローマ字表示" }
]
```

`romaji` は `kana` をローマ字読みしたもの（ヒント表示用）。

## kana フィールドの絶対ルール

1. **漢字・カタカナは禁止**。必ずひらがなに変換する
   - NG: `kana: "東京"` / OK: `kana: "とうきょう"`
   - NG: `kana: "コーヒー"` / OK: `kana: "こーひー"`
2. **英単語は小文字のみ**。大文字禁止
   - NG: `kana: "USB接続"` / OK: `kana: "usbせつぞく"`
   - NG: `kana: "Wi-Fi"` / OK: `kana: "wi-fiがつながらない"`
3. **長音符 `ー` はひらがなと共存 OK**
   - `こーひー`、`すぽーつ` のように使う
4. **完全英語文は禁止**。必ず日本語を含む
5. **鍵カッコ「」『』は禁止**
6. **kana の長さは最大 29 文字**（JavaScript の `s.length`（UTF-16 code unit）で数える。`ー` `・` 半角英数も 1 文字）
7. **スペース禁止**
8. **全角英数字禁止**（半角のみ）

## 数字・英字混在の例

| jp | kana |
|---|---|
| USB接続する | usbせつぞくする |
| 3時間勉強した | 3じかんべんきょうした |
| Wi-Fiが繋がらない | wi-fiがつながらない |

## ん のルール

- `ん` の次が母音・y・n・語末の場合 → `nn` で打つ必要があるが、kana としては `ほんをよむ` と書けばエンジンが自動判定する

## 多様性の要件（必須）

生成する文は、**内容・カテゴリ・文型をできるだけバラけさせること**。以下のカテゴリから偏りなく選ぶ：

- 日常生活（朝の準備、歯磨き、洗濯、掃除、ゴミ出しなど）
- 食べ物・飲み物（料理名、味の感想、食事シーン、お菓子、飲み物など）
- 自然・季節（天気、桜・紅葉、海・山、動物、虫、星空など）
- スポーツ・運動（競技名、練習、筋トレ、ウォーキング、体の感覚など）
- 技術・IT（PC操作、スマートフォン、アプリ、SNS、インターネットなど）
- 学習・教育（勉強、読書、試験、授業、資格、図書館など）
- 仕事・ビジネス（会議、メール、締め切り、プレゼン、残業など）
- 感情・心理（喜び、驚き、悩み、緊張、安心、後悔など）
- 旅行・観光（交通、ホテル、名所、土産、パスポートなど）
- 買い物・お金（スーパー、値段、セール、支払い、節約など）
- 健康・医療（病院、薬、睡眠、ダイエット、体調など）
- 家族・人間関係（友人、恋愛、家族の会話、近所付き合いなど）
- 趣味・娯楽（映画、音楽、ゲーム、漫画、カラオケ、写真など）
- 乗り物・交通（電車、バス、車、自転車、渋滞、時刻表など）
- 天気・気候（晴れ、雨、台風、雪、気温、湿度など）
- 料理・レシピ（材料、調理手順、火加減、味付けなど）
- 動物・ペット（犬、猫、魚、鳥、飼育、散歩など）
- ことわざ・慣用句（日本語のことわざや慣用句を文に含める）

<COUNT>文生成する場合、同じカテゴリから連続して2文以上取らないこと（必ず次は別カテゴリにする）。また、文の構造（〜する、〜だ、〜ている、〜たい、〜かもしれない、〜てしまった、〜らしい、疑問文、命令文など）も繰り返さず多様にすること。

## テーマ

<THEME_INSTRUCTION>

## チェックリスト（出力前に各文を確認）

- [ ] kana に漢字が含まれていないか
- [ ] kana にカタカナが含まれていないか（ー・は OK）
- [ ] kana に大文字 ASCII が含まれていないか
- [ ] jp と kana の対応が正しいか（英字部分は英字、日本語部分はひらがな）
- [ ] 鍵カッコが含まれていないか
- [ ] kana が 29 文字以内か
- [ ] 完全英語文でないか
```

**プレースホルダーの置き換え:**
- `<COUNT>` → ユーザーが指定した生成数
- `<LENGTH_INSTRUCTION>` → 以下のいずれか（範囲はすべて両端を含む / `s.length` で計測）:
  - 短め: `kana の長さ s.length は 1〜15 にしてください（両端を含む）`
  - 普通: `kana の長さ s.length は 16〜22 にしてください（両端を含む）`
  - 長め: `kana の長さ s.length は 23〜29 にしてください（両端を含む）`
- `<THEME_INSTRUCTION>` → ユーザーが指定したテーマ（未指定なら `日常的な日本語の文にしてください`）

生成エージェントの出力（JSON文字列）を受け取る。

## ステップ 3: JSON を一時ファイルに保存

生成エージェントが返した JSON を一時ファイルに書き出す。一時ファイル先は以下の順で決める。

1. **第一候補**: `/tmp/sentences_draft.json`
2. **フォールバック発動条件**: 第一候補 `/tmp/sentences_draft.json` への書き込み試行（Write tool / Bash 経由の `tee` 等いずれでも可）が `permission denied` / `アクセスは禁止されています` 等のエラーで弾かれたら、出力先と同じディレクトリに隠しファイル `.sentences_draft.json` を置く。例: 出力先が `output/sentences.json` なら `output/.sentences_draft.json`。**事前に環境を判定せず、まず `/tmp` を試して失敗してから切り替える**。

書き出し後、検証スクリプトには **実際に書き出したパス** を渡す（次ステップのコマンド例の `<draft_path>` 部分）。

## ステップ 4: 検証スクリプトを実行

```bash
node --experimental-strip-types scripts/validate-sentences.ts <draft_path>
```

Bash ツールは exit code ≠ 0 のときエラーとして表示される（`Error: ... (exit code N)` の形）。エラー表示が無ければ `0`、有る場合は表示された数値を見る。`echo $?` 等の後続コマンドは不要。

スクリプトの終了コードを確認する：
- `0` → 全文有効。ステップ 5 へ
- `1` (type-A エラーのみ) → type-A エラー文を除外してステップ 5 へ
- `2` (type-B エラーのみ) → type-B エラーについて GitHub Issue を登録（ステップ 6）してステップ 5 へ
- `3` (両方) → type-A 除外 + type-B Issue 登録の両方を行いステップ 5 へ

## ステップ 5: 有効な文を出力先にマージして書き出す

検証をパスした文のみを、ユーザーが指定した出力先に**追記（マージ）**する。

手順：
1. 出力先ファイルが**すでに存在する場合**は、そのJSONを読み込んで既存の配列を取得する
2. 既存配列と新規文の配列を結合（`[...既存, ...新規]`）する
3. 結合した配列全体をファイルに書き出す（上書き）

出力前にディレクトリが存在しない場合は `mkdir -p` 等で作成する。

ファイルに書き出し後、結果をユーザーに報告する：
- 生成数 / 有効数 / type-A 除外数 / type-B Issue 登録数
- 追記前の既存件数 / 追記後の合計件数
- 出力先パス

## ステップ 6: type-B エラーの GitHub Issue 登録

type-B エラー（エンジン側のバグ）が存在する場合、各エラーについて GitHub Issue を登録する。

**リポジトリ: `ahy4/typing`**

Issue の内容：
- **タイトル**: `[typing-engine] parseKana が処理できない kana パターン: "<kana文字列>"`
- **本文**:
  ```
  ## 概要
  お題生成スキルの検証で、有効な kana 文字列を parseKana() が正しく処理できないケースを検出しました。

  ## 再現情報
  - jp: <japanese文字列>
  - kana: <kana文字列>
  - エラー詳細: <スクリプトが出力したエラー理由>

  ## 期待される動作
  上記 kana を parseKana() に渡すと、全セグメントに有効な romaji オプションが返るべきです。

  ## 発生環境
  scripts/validate-sentences.ts (type-B error)
  ```
